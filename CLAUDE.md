# NDT/Colcheck PDF Data Extraction Pipeline

## Project Purpose

This project extracts structured data from NDT (Non-Destructive Testing) / Colcheck PDF inspection reports for street lighting columns. The pipeline reads PDF files, converts text to JSON, then generates formatted Excel spreadsheets for analysis.

## Domain Context

The PDFs are rPec inspection reports generated by Equans (contract reference `0629_001_Equans`). Each PDF covers a single street lighting column and contains:

- **Column identity**: Central Asset Id, Column Name, Street, Id, Road Number, Ward
- **Column characteristics**: Column type (e.g., CD - Decorative), height, material (e.g., S1 - Steel), fixing type, heritage status, listed status, commission date, coordinates (OSE/OSN)
- **Condition indices**: Condition rating (e.g., Good), CCIav, CCIcrt, ARL (years) — ARL can be a number or `--` when not applicable
- **Thickness/loss measurements**: Thickness Max/Min/Ave (mm), % Loss Max, % Loss Ave (PL)
- **Test dates**: Test date, Next test date
- **GN22 element assessments**: 9 structural elements (base external, base internal, door, embellishment, foundation, lantern, shaft external, shaft internal, shoulder) — each with Severity grade (single letter, e.g., B/D/E), Extent (number), and a text Comment
- **Inspector comments**: Free-text notes prefixed with `- `, found at the end of the data (e.g., monitoring instructions for leaning columns)

## Pipeline Overview

The pipeline runs as two sequential Python scripts:

### Step 1: `pdf_to_json.py` — PDF to JSON extraction

- Scans the script's directory for all `*.pdf` files
- Uses **PyMuPDF** (`fitz`) to extract text-only content from every page (images are ignored)
- Outputs a single `output.json` keyed by filename, with per-page text arrays
- All meaningful data lives on **page 1** of each PDF; pages 2+ are typically headers or image-only pages

### Step 2: `json_to_excel.py` — JSON to Excel conversion

- Reads `output.json` and parses the raw page 1 text into structured fields
- Text parsing handles three patterns:
  - **Next-line fields**: label on one line, value on the next (most fields)
  - **Inline fields**: value embedded on the same line as the label (`Thickness Max (mm) 6.90`, `Thickness (mm) Min 6.81`)
  - **GN22 element blocks**: element name followed by severity, extent, and multi-line comment
- GN22 element comments run until the next known element name or section boundary
- Inspector comments (lines starting with `- `) are joined with ` | ` separators
- Outputs `asset_report.xlsx` with 53 columns, styled headers, frozen top row, auto-fitted widths

## Dependencies

```
pip install pymupdf openpyxl
```

- **pymupdf** (imported as `fitz`) — PDF text extraction
- **openpyxl** — Excel file creation and formatting

Python 3.10+ required (uses `list[str]` type hint syntax).

## How to Run

```bash
# Step 1: Extract text from all PDFs into output.json
python pdf_to_json.py

# Step 2: Parse JSON and generate asset_report.xlsx
python json_to_excel.py
```

Both scripts auto-detect their working directory (same folder as the script). Drop new PDFs into the folder and re-run both scripts to include them.

## JSON Structure (output.json)

```json
{
  "filename.pdf": {
    "filename": "filename.pdf",
    "total_pages": 8,
    "pages": [
      { "page_number": 1, "text": "raw extracted text..." },
      { "page_number": 2, "text": "..." }
    ]
  }
}
```

## Excel Column Layout (asset_report.xlsx)

53 columns in this order:

1. **Identity** (6 cols): Central Asset Id, Column Name, Id, Street, Road Number, Ward
2. **Characteristics** (8 cols): Column Type, Column Height, Material, Fixing Type, Heritage Y/N, Listed Status, Commission Date, OSE/OSN
3. **Condition** (4 cols): Condition, CCIav, CCIcrt, ARL (years)
4. **Measurements** (5 cols): Thickness Max/Min/Ave (mm), % Loss Max, % Loss Ave
5. **Test Dates** (2 cols): Test Date, Next Test
6. **GN22 Elements** (27 cols): 9 elements x 3 cols each (Severity, Extent, Comment)
7. **Comments** (1 col): Inspector notes

## Parsing Details and Edge Cases

- **ARL field**: Can be a number (e.g., `19`) or `--` when not calculable. Both are stored as strings.
- **Inline thickness fields**: `Thickness Max (mm)` and `Thickness (mm) Min` have values on the same line in the PDF text, extracted via regex. `Thickness Ave (mm)` follows the normal next-line pattern.
- **`Thickness (mm) Min` renaming**: The PDF label is `Thickness (mm) Min` but it's displayed as `Thickness Min (mm)` in Excel for consistency with Max/Ave.
- **Multi-line GN22 comments**: Some comments span multiple lines (e.g., the `[AMBER]` foundation note on Camden Hill Road). The parser reads lines until it hits the next element name or a single uppercase letter (next element's severity grade).
- **Inspector comments**: Lines starting with `- ` anywhere in the text are collected as general comments, joined with ` | `.
- **Empty pages**: Pages 2+ often contain only images or a brief header. The JSON stores them but the Excel script only uses page 1.

## Coding Conventions

- Type hints on all function signatures
- Docstrings on all functions (Google style with Args/Returns)
- Step-by-step comments documenting each stage
- Constants defined at module level (no magic strings in logic)
- `pathlib.Path` for all file operations
- `if __name__ == "__main__"` guard on all scripts
- Error handling per-item (one bad PDF/asset doesn't stop the batch)
- Print-based progress logging
